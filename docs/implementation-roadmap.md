## 2. Sprint 规划

### Sprint 1（W1-W2）：地基 + 合约 + 用户体系

**Sprint 目标：** 合约部署测试网 + 用户注册→资料→认证全链路可用

| 交付物 | 负责人 | 验收标准 |
|--------|--------|---------|
| Next.js项目 + 深色主题 + UI组件库 | 小code | `pnpm dev` 启动，组件库风格统一 |
| Supabase Schema + 种子数据 | 小code + 小research | 全部表创建，5个预设用户+3个任务 |
| Privy集成 + Base Sepolia配置 | 小code | 邮箱登录→控制台输出钱包地址 |
| **Privy国内可用性实测** ⚠️ | 小ops | 3个测试点(北京/上海/深圳)P95<3s |
| BountyEscrow合约 + 安全加固 | 小code | forge coverage ≥95%，Slither 0 High/Medium |
| ReputationSBT + EscrowFactory | 小code | SBT不可转让，Factory创建+追踪正常 |
| 合约部署Base Sepolia + ABI导出 | 小code | BaseScan可查，前端类型就绪 |
| 注册/登录 + 用户API | 小code | 注册→JWT→资料编辑→保存 |
| 个人资料页 + 精英列表/详情/搜索 | 小code | 搜索中英文、筛选实时过滤 |
| VERIFIED认证申请流程 | 小code | 申请→写入DB→管理员审核 |
| 连接请求功能 | 小code | 发送/接受/拒绝 |

**Sprint 1 Go/No-Go 检查点（Day 7）：**
- ✅ Privy国内P95延迟 < 3s → 继续
- ❌ Privy不可用 → 启动Plan B（切换Dynamic/MetaMask直连）
- ✅ 合约Slither 0个High/Medium → 继续部署测试网
- ❌ 发现严重漏洞 → +3天重写，Sprint 2顺延

### Sprint 2（W3-W4）：任务池核心闭环 ⭐ 关键路径

**Sprint 目标：** 任务发布→接单→Escrow托管→交付→验收→链上结算 全链路跑通

| 交付物 | 负责人 | 验收标准 |
|--------|--------|---------|
| 任务API（CRUD+状态+权限） | 小code | 状态只能向前流转，权限校验正确 |
| 任务发布页（表单+USDC存入Escrow） | 小code | 发布→Approve+Transfer→Escrow锁定→BaseScan可查 |
| 任务列表（筛选Pills+搜索+排序） | 小code | 分类筛选实时，分页滚动加载 |
| 任务详情（Markdown+链上状态） | 小code | 代码高亮，Escrow金额实时显示 |
| 认领申请（申请→审核→确认） | 小code | VERIFIED才能申请，合约状态Open→InProgress |
| 交付提交（工程师端） | 小code | 交付说明+链接→状态Delivered |
| 验收确认（发布方端） | 小code | 确认→合约approve()→90%+10%释放→Completed |
| **自动验收机制** ⚠️ | 小code | 交付7天未操作→自动验收→资金释放 |

| 超时取消+退款 | 小code | 宽限期后一键退款→Cancelled |

**Sprint 2 Go/No-Go 检查点（Day 14 / W4 EOD）：**
- ✅ 端到端跑通一笔任务（BaseScan交易证明）→ 进入Phase 4
- ❌ 合约交互有阻断Bug → 优先修复，Phase 4延后2天

- ⚠️ USDC精度处理有误 → 必须修，不进入下阶段

## 5. 技术债务预算

### 5.1 允许的技术债（MVP范围）

| 技术债 | 允许程度 | 影响 | 清理时机 |
|--------|---------|------|---------|
| API Routes单体（不拆微服务） | ✅ 完全允许 | 2000 MAU内无问题 | v2.0 |
| 无自动化CI/CD | ✅ 允许 | 手动部署可接受 | v1.0上线后Week 7 |
| Supabase RLS替代业务层权限 | ✅ 允许 | 简单但调试成本高 | v1.1 |
| 合约无Proxy（不可升级） | ✅ 允许 | 修复需重新部署 | 审计通过后加Proxy |
| 无Rate Limiting | ⚠️ 临界允许 | 初期流量小 | v1.0上线前必须补上 |
| 前端无单元测试 | ✅ 允许 | E2E测试兜底 | v1.1 |
| 硬编码配置（链ID/合约地址） | ⚠️ 允许 | 多链部署时需重构 | v1.1 |

### 5.2 不允许的技术债（红线）

| 技术债 | 原因 |
|--------|------|
| ❌ 合约不经过Slither扫描就部署 | 管理真实资金，安全是底线 |
| ❌ 不做任何错误处理就上线 | 用户遇到白屏=永久流失 |
| ❌ 跳过Privy国内实测 | 可能整个钱包方案不可用 |
| ❌ 不做5人可用性测试 | UX是冷启动成败关键 |
| ❌ USDC金额用number不用BigInt | 精度问题会导致资金错误 |

### 5.3 还债计划

| 时间 | 还债内容 | 工时 |
|------|---------|------|
| **Week 7** | CI/CD自动化 + Rate Limiting | 8h |
| **Week 8-9** | 合约加TransparentProxy + 审计修复 | 16h |
| **v1.1 (Week 9-12)** | Supabase查询优化 + 前端单测 + 配置抽象 | 24h |
| **v2.0** | 微服务拆分评估 + Ponder链上索引 | 40h |

## 6. 回滚方案

### 6.1 Phase级别回滚

| 失败场景 | 触发条件 | Plan B | 额外工期 |
|---------|---------|--------|---------|
| **Privy国内不可用** | G1 No-Go（P95>3s或无法连接） | → 切换Dynamic SDK（API兼容，迁移2天）→ 如Dynamic也不行→MetaMask直连（3天） | +3-5天 |
| **合约审计发现严重漏洞** | G2/G5 No-Go（High级别漏洞） | → 重写漏洞模块+补充测试→重新审计 | +5-7天 |
| **Base链严重故障** | Sequencer长时间停机 | → 部署到Arbitrum One（EVM兼容，1天切换）→ 合约地址配置化，前端切RPC | +1-2天 |
| **USDC流动性问题** | 极低概率 | → 评估USDT on Base作为备选 | +3天 |
| **冷启动失败** | 上线2周<100注册或<20任务 | → 加大种子任务到100个+KOL定向→考虑开放VERIFIED标准→评估pivot方向 | 运营调整 |

### 6.2 功能级别降级

| 功能 | 完整版 | 降级版 | 降级条件 |
|------|--------|--------|---------|
| 钱包 | Privy MPC嵌入式 | MetaMask直连 | Privy不可用 |
| 支付 | USDC Escrow自动托管 | 线下确认+手动转账 | 合约部署失败 |
| 信誉 | SBT自动铸造+展示 | DB记录+前端展示 | SBT合约有问题 |
| 通知 | 实时站内推送 | 轮询刷新 | WebSocket不可用 |
| 搜索 | 全文搜索+筛选 | 仅筛选（无搜索） | 性能不达标 |
| 任务流程 | 完整状态机 | 简化为发布→接单→完成 | 验收流程有Bug |

### 6.3 核心降级路径（最坏情况）

```
完整MVP (Escrow+SBT+Privy+Base)
  ↓ 降级1: Privy不可用
MVP-MetaMask (MetaMask直连替代Privy)
  ↓ 降级2: 合约有安全问题
MVP-Lite (DB信誉代替SBT，线下转账代替Escrow)
  ↓ 降级3: Base链故障
MVP-Arbitrum (一键切到Arbitrum)
  ↓ 最低可用
核心闭环: 发布任务 → 认领 → 交付 → 确认（无链上，纯DB驱动）
```

> **原则：** 即使区块链完全不可用，平台仍可降级为中心化模式运行。链上是增强，不是依赖。

## 7. 关键路径与并行策略

```
         ┌── Phase 1 (合约: W1) ──────────────────────┐
Phase 0 ─┤                                              ├── Phase 3 (任务池: W3-W4) ── Phase 4-5 (W5-W6)
(W1前半) └── Phase 2 (用户: W1后半-W2) ───────────────┘
```

**并行机会：**
- W1 Day 3-5：合约开发 ∥ 用户系统前端（小code优先合约，合约完成后切前端）
- W2：合约测试/部署 ∥ 用户系统API+页面
- W5：小code打磨功能 ∥ 小ops E2E测试 ∥ 小research准备种子任务

**关键瓶颈：** 小code是全程瓶颈（224h），需严格控制MVP范围不蔓延。
